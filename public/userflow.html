CodeEthnics — User Flow

    
From first visit → email verification → sign-in → role-based panel


    


      
      

        

          
1

          
Student Registers

        

        

          A new user submits their name, email, and password. The role is
          always forced to student by the server — it cannot be set by the client.
        

        

          POST
          /auth/sign-up
        

        

          

            
Request Body

            
{
  "name": "Jane Doe",
  "email": "jane@uni.edu",
  "password": "SecurePass123"
}

          

          

            
Response 200

            
{
  "user": {
    "id": "...",
    "email": "jane@uni.edu",
    "emailVerified": false,
    "role": "student"
  }
}

          

        

        

          📧 A verification email is sent automatically via Nodemailer.
          The session is NOT yet active — user must verify first.
        

      


      


      
      

        

          
2

          
Verify Email Address

        

        

          The user clicks the link in their inbox, which hits the backend verify endpoint.
          On success, the email is marked verified and the user is auto-signed-in.
        

        

          GET
          /auth/verify-email?token=<token>
        

        

          

            
Email Link

            
http://localhost:3000
/verify-email
?token=eyJhbGci...

          

          

            
On Success

            
{
  "user": {
    "emailVerified": true
  },
  "session": { ... }
}
→ auto-signed in

          

        

        

          ⏰ Token expires in 1 hour.
          If expired, the user can request a new OTP via POST /auth/email-otp/send-verification-otp.
        

      


      


      
      

        

          
3

          
Sign In

        

        

          Verified user submits credentials. Rate-limited to 8 requests/min per IP.
          A session cookie (better-auth.session_token) is set automatically.
          All previous sessions are deleted (single-device enforcement).
        

        

          POST
          /auth/sign-in
        

        

          

            
Request Body

            
{
  "email": "jane@uni.edu",
  "password": "SecurePass123"
}

          

          

            
Response 200

            
{
  "session": {
    "token": "...",
    "expiresAt": "7 days"
  },
  "user": {
    "role": "student"
  }
}

          

        

        

          

            
Error — Not Verified (403)

            
{ "message": "Email not verified" }

          

          

            
Error — Wrong Password (401)

            
{ "message": "Invalid credentials" }

          

        

      


      


      
      

        

          
4

          
Frontend Reads Session & Role

        

        

          After sign-in the frontend calls /me to get user info and the redirect path,
          or /auth/get-session to get the full session object.
          The session cookie is sent automatically on every request.
        

        

          

            
GET /me

            
{
  "user": {
    "id": "...",
    "email": "jane@uni.edu",
    "role": "student",
    "emailVerified": true
  },
  "redirect": "/student/dashboard"
}

          

          

            
GET /auth/get-session

            
{
  "user": { ... },
  "session": {
    "id": "...",
    "expiresAt": "...",
    "ipAddress": "..."
  }
}

6

          
Access Protected Panel

        

        

          Every protected route runs through middleware that checks the session cookie first,
          then optionally checks the user's role. Session is valid for 7 days, auto-refreshed every 24h.
        


        

          

            🔒 Route Protection Matrix
          

          

            

              GET
              /student/dashboard
              requireAuth
              student role
            

            

              GET
              /admin/dashboard
              requireAuth
              product_admin
            

            

              POST
              /admin/create-user
              requireAuth
              product_admin
            

            

              GET
              /admin/users
              requireAuth
              product_admin
            

            

              GET
              /me
              requireAuth
              any role
            

          

        


        

          🚪 Sign-out via POST /auth/sign-out immediately deletes the session from DB.
          Next request to any protected route returns 401 Unauthorized.
